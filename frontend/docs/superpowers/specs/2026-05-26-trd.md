## Resumen Técnico

CobroDetector es una aplicación web full-stack construida con **Next.js 14 App Router** desplegada en **Vercel**. Utiliza **Supabase** como backend (PostgreSQL + Auth + Storage + RLS), **Groq SDK** para detección de anomalías con IA (Llama 3.1 8B), y **MercadoPago** para procesamiento de pagos. El rate limiting se implementa con **Upstash Redis**.

### Decisiones Tecnológicas Clave

| Decisión | Justificación |
|----------|---------------|
| **Next.js 14 App Router** | Server Components para SEO y rendimiento, API Routes en el mismo proyecto, Vercel-native |
| **Supabase** | PostgreSQL gestionado, Auth integrada, RLS para seguridad a nivel de fila, Storage para archivos |
| **Groq (Llama 3.1 8B)** | Inferencia rápida (~2s), bajo costo, JSON mode nativo, sin necesidad de fine-tuning |
| **MercadoPago** | Líder en pagos en Chile, SDK oficial, webhooks, soporte para cuotas |
| **Upstash Redis** | Serverless Redis, sin cold starts, HTTP-based (compatible con Vercel Edge) |
| **Tailwind CSS** | Utility-first, zero-runtime, tree-shaking automático, tema oscuro nativo |
| **TypeScript** | Tipado estricto, ESLint sin `any` (excepto tipos DB), zod para validación runtime |

## Decisiones Técnicas Clave

### 1. Row Level Security (RLS)

**Decisión**: Todas las tablas usan RLS con políticas `auth.uid() = user_id`.

**Justificación**: Seguridad a nivel de base de datos. Incluso si un bug en la aplicación expone una query, RLS garantiza que un usuario solo accede a sus propios datos.

**Implementación**: 12 tablas con RLS, 38 políticas (SELECT/INSERT/UPDATE/DELETE). Las políticas usan `(SELECT auth.uid()) = user_id` para evitar evaluar `auth.uid()` por cada fila.

### 2. Funciones SECURITY DEFINER en Schema Internal

**Decisión**: Funciones que necesitan bypass RLS (verify_api_key, consume_credit) residen en schema `internal` privado. El schema `public` tiene wrappers SECURITY INVOKER que usan `auth.uid()`.

**Justificación**: Separación de privilegios. El schema `internal` no es accesible vía REST API. Las funciones privilegiadas no pueden ser llamadas directamente desde el frontend.

### 3. Rate Limiting con Graceful Fallback

**Decisión**: Upstash Redis para rate limiting, con fallback a `{ success: true }` si Redis no está configurado.

**Justificación**: No bloquear la aplicación si Redis falla o no está configurado. La seguridad es importante pero la disponibilidad también.

### 4. CSRF Protection en Logout

**Decisión**: Validar `origin`/`referer`/`host` en el endpoint de logout contra una whitelist de dominios autorizados.

**Justificación**: Prevenir que un sitio malicioso fuerce el logout del usuario vía CSRF.

**Whitelist**: `localhost`, `cobrodetector.cl`, `*.vercel.app`.

### 5. CSP Headers

**Decisión**: Content-Security-Policy con `'unsafe-inline'` en `script-src`.

**Justificación**: Requerido por Next.js para scripts inline de hidratación. Sin esto, las páginas de login rompen en producción.

### 6. Parseo Multi-Formato

**Decisión**: Soportar CSV (nativo), Excel (exceljs), y PDF (pdf-parse).

**Justificación**: Los bancos chilenos entregan estados de cuenta en diversos formatos. Santander usa PDF principalmente, otros bancos usan Excel o CSV.

### 7. Workaround Bug @supabase/ssr

**Decisión**: Helper `tables(client)` en `src/lib/supabase/db.ts` que usa `as any` para bypassear el bug de inferencia de tipos.

**Justificación**: `@supabase/ssr` v0.10.3 tiene un bug que hace que las queries de tabla retornen `never`. El workaround permite queries tipadas sin esperar el fix upstream. Se deshabilitó `ignoreBuildErrors` en TypeScript como compensación.

## API Design

### Principios

- **Auth explícita**: cada handler de API es responsable de su propia autenticación. El middleware solo protege rutas de página.
- **Validación Zod**: todos los inputs se validan con Zod. Los errores solo exponen el primer issue (no el array completo).
- **Rate limiting**: endpoints sensibles (analyze, health, auth) tienen rate limiting por IP.
- **Errores genéricos**: `ApiError` expone mensajes controlados, no stack traces.

### Endpoints

| Método | Ruta | Auth | Rate Limit | Descripción |
|--------|------|------|------------|-------------|
| `POST` | `/api/analyze` | Sesión | 3 req/60s | Upload + análisis |
| `DELETE` | `/api/analyses/[id]` | Sesión | - | Eliminar análisis |
| `POST` | `/api/payments/create` | Sesión | - | Crear preferencia MP |
| `POST` | `/api/payments/unlock-report` | Sesión | - | Pagar 20% Platino |
| `GET/POST` | `/api/companies` | Sesión | - | Listar/Crear empresa |
| `GET/PATCH/DELETE` | `/api/companies/[id]` | Sesión | - | CRUD empresa |
| `POST` | `/api/documents/complaint-letter` | Sesión | - | Generar DOCX |
| `POST` | `/api/documents/complaint-letter/pdf` | Sesión | - | Generar PDF |
| `POST` | `/api/logout` | Sesión | - | Logout CSRF-safe |
| `POST` | `/api/auth/check-rate-limit` | Pública | - | Rate limit check |
| `GET` | `/api/health` | Pública | 3 req/60s | Health check |
| `POST` | `/api/payments/webhook` | HMAC | - | Webhook MP |
| `GET` | `/api/v1/analyses/[id]` | API Key | Por key | API pública |

### Versionado de API

La API v1 usa el prefijo `/api/v1/` y autenticación por API Key (Bearer token). Las rutas internas no están versionadas (son parte de la aplicación, no una API pública).

## Seguridad

### Capas de Seguridad

```
Capa 1: Transporte
  └── HTTPS (TLS 1.3) en todo el tráfico

Capa 2: Aplicación
  ├── CSP Headers (X-Content-Type-Options, X-Frame-Options, Referrer-Policy)
  ├── Rate Limiting (Upstash Redis)
  ├── CSRF Protection (logout)
  └── Input Sanitization (Zod)

Capa 3: Autenticación
  ├── Supabase Auth (sesiones JWT en cookies)
  ├── API Keys (SHA-256 hash + RPC verification)
  └── Webhook MercadoPago (HMAC-SHA256 + timingSafeEqual)

Capa 4: Base de Datos
  ├── Row Level Security (12 tablas, 38 políticas)
  ├── Schema Internal privado (funciones SECURITY DEFINER)
  ├── search_path fijo en funciones privilegiadas
  └── Grants mínimos (anon solo puede verify_api_key)

Capa 5: IA
  └── Anti Prompt-Injection (sanitizeDescription, límite 200 chars)
```
